# US-02 · Access to oOh!media investor materials

**Priority:** P0 · **Story points:** 2

## Story

> As the assistant, I need access to oOh!media's public investor materials so that I can answer questions grounded in real company documents.

## Acceptance criteria (verbatim from `Candidate_04_User_Stories.docx`)

1. The assistant can answer questions about content found in oOh!media annual and half year reports.
2. The assistant can answer questions about content found in investor presentations.
3. Each answer identifies which document the information came from, including the document title and type.
4. The materials are sourced from the oOh!media investor centre at https://investors.oohmedia.com.au/investor-centre/.
5. The selection of materials is justifiable given the assessment timebox, covering at least two different document types and two different reporting periods.
6. Adding new documents in the future does not require rebuilding the system from scratch.

## Why this matters

This is the knowledge layer. Every grounded answer in US-03 and US-05 depends on chunks produced here carrying honest provenance. The Solution Design says explicitly: "It does not need to ingest the entire archive. What matters is that the materials it uses are traceable and that the selection is justifiable."

## Definition of done

- A small, justifiable set of PDFs lives under `data/pdfs/` — at minimum two document types (annual/half-year report **and** investor presentation) across two reporting periods. Source them from the oOh!media investor centre and record the source URL in each `Citation`.
- `core/ingest.py` exposes an idempotent `ingest(pdf_dir)` entry point that can be re-run to add new PDFs without wiping the collection (US-02 AC6).
- PDFs are parsed with pdfplumber. Each chunk carries, at minimum: `source_id`, `doc_title`, `doc_type`, `period`, `page`, `snippet`, `url`. These map 1:1 to the `Citation` schema in `core/schema.py`.
- Chunks are embedded with OpenAI `text-embedding-3-small` and written to a persistent ChromaDB collection at `data/chroma/`.
- Ingest can be invoked from the CLI: `uv run python -m core.ingest data/pdfs`.
- The `data/pdfs/` directory includes a small `SOURCES.md` listing each file, its investor-centre URL, type, and period — this is your justification for AC5.

## Verification

```bash
source .venv/bin/activate
uv run python -m core.ingest data/pdfs
uv run python -c "from core.retrieval import search; \
  [print(c.doc_title, c.doc_type, c.period, c.page) for c in search('revenue', k=5)]"
```

Confirm:

1. The ingest command completes without errors and prints a count of chunks written.
2. Running ingest a second time does not duplicate chunks (idempotency — check with a count query).
3. The search output shows real document titles, types, periods, and page numbers — no `None`, no placeholders.
4. `data/pdfs/SOURCES.md` lists at least two types and two periods with working investor-centre links.
